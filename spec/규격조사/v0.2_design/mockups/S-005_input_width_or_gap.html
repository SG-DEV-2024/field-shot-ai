입력 검증 (실시간)

  • 시작값 / 끝값 · 둘 다 숫자(소수 허용)여야 함. 빈 값이면 측정값 미계산.
  • 끝값 > 시작값 · 동일 또는 작으면 끝값 입력란 빨간 외곽선 + "끝값은 시작값보다 커야 합니다" 메시지 + 자동 계산 카드 회색 처리
  • 측정값 자동 계산 · measured_value = end_value - start_value · 단위 mm
  • 저장 버튼 활성 조건 · 두 값 모두 유효 + 끝값 > 시작값. 그 외엔 회색 disabled
  • 비고 는 선택 입력 — 저장 활성 조건에 영향 없음

폭 vs 간격 — 시작값 차이

두 subtype은 같은 화면을 사용하지만, 작업자가 입력하는 시작값의 의미가 다름.

subtype시작값끝값의미
wall_or_member_width (폭)일반적으로 0부재 사이즈줄자 0이 부재 시작면에 정확히 닿음
member_gap (간격)임의 값 (예: 120)임의 끝 눈금 (예: 420)줄자 임의 눈금에서 시작 — 부재 모서리 손상·접근 어려움 등의 케이스
  • 측정값 계산식은 동일 · 두 subtype 모두 end - start. 폭은 start=0이라 결국 end만큼이 됨
  • placeholder는 동일 · "줄자 시작 눈금" / "줄자 끝 눈금"으로 표시 (구체 숫자 예시 안 함 — 학습 다양성)
  • 학습 데이터 분리 · subtype 메타데이터로 저장되어 후처리에서 폭/간격 케이스 분리 가능

학습 품질 수집 — 비고 자유 기입

v0.2 결정 (2026-05-29): 작업자에게 6개 체크박스+칩 자가 체크는 입력 부담만 큰 데 비해 일관성이 낮다고 판단, quality_flags 자가 체크 UI를 제거하고 비고 자유 기입으로 통합.

  • 이상 항목 자유 기입 · 비고 placeholder에 예시("줄자 흐림, 가림, edge 불명확 등") 노출해 자유 텍스트로 수집
  • 구조화 메타데이터 · 학습 단계의 별도 큐레이터/검수자가 텍스트 비고를 기반으로 라벨링하거나, 이미지 분석 모델로 자동 추출
  • 현장 입력 부담 최소화 · 정상 데이터는 비고 비워두고 빠르게 통과, 이상이 있을 때만 작성
  • annotation 저장 단계의 qualityFlags 필드 유지 · 후처리에서 채워질 수 있도록 모델 구조는 그대로. UI 입력만 없앰

키보드 처리 정책 (S-005 · S-008 공통)

숫자/텍스트 입력 시 모바일 키패드가 화면 하단을 가리지 않도록 다음 정책을 적용한다.

  • Scaffold resizeToAvoidBottomInset: true · v1 데이터 입력 화면과 동일. 키보드 등장 시 body가 축소되어 하단 저장 버튼이 키보드 위로 자동 이동
  • scroll-area 하단 buffer 120px · 비고 textarea를 키보드 위로 끌어올릴 여유 공간 확보. Android 숫자 키패드(~280-330px) 기준 안전치
  • 포커스 input 자동 스크롤 · Flutter TextField 기본 동작(EnsureVisibleWhenFocused). 시작/끝 input 탭 시 자동으로 가시 영역으로 이동
  • textarea 활성 시 사용자 스크롤 권장 · 비고 textarea(140px)는 키보드 위쪽 영역에 표시될 수 있도록 사용자가 손가락으로 위로 스와이프 가능. buffer 120px이 이를 가능케 함
  • iOS 대응 · iOS 숫자 키패드도 유사한 높이. 동일 정책으로 충분
